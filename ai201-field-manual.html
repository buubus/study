00  Orientation

What the exam is, and where the marks live.

The Agentforce Specialist certification validates that you can design, configure, test and deploy autonomous AI agents on the Salesforce Platform. It is scenario-heavy: most questions hand you a business situation and ask which agent, topic, action, template, retriever or deployment step fits. Rote definitions earn little; recognising the shape of a problem earns almost everything.

Five weighted domains. The weighting is the single most important study input — more than half the exam sits in the first two.

Naming note

As of 14 October 2025, Data Cloud was rebranded Data 360. The functionality is unchanged and you will still see "Data Cloud" in parts of the UI and documentation. The exam treats the terms interchangeably.

01  Domain · 35%

AI Agents.

Everything in this domain hangs off one mental model: an agent is a container of topics, topics are containers of actions, and Atlas is the reasoning engine that decides which topic and action to fire. Hold that hierarchy and most scenario questions resolve themselves.

Atlas — the reasoning engine

Atlas is the LLM-driven brain. On each turn it runs classify → plan → act → respond: it reads the utterance, matches it to a topic using that topic's scope and instructions, selects one or more actions, executes them, then composes the reply. When an answer is wrong, diagnose top-down.

Topic routing→Action selection→Action logic

The other line the exam draws: Agentforce agents are autonomous and goal-driven, unlike the older assisted, user-driven Copilot experience. Autonomy is the differentiator.

Topics versus actions

  • A topic is a job the agent can do — a bounded responsibility defined by scope, instructions, and its available actions. Good instructions are specific about what the agent should and should not do; vague instructions are the classic wrong answer.
  • An action is a concrete capability. Standard actions ship with Salesforce (Identify Record Data, Query Records, Knowledge answers). Custom actions are built on Flow, Apex, or a Prompt Template — memorise that trio, because "integrate with an external ERP" expects Flow or Apex, not a prompt.

Agent types

TypeWhen it applies
Service AgentCustomer-facing service and support; runs on channels with no logged-in user.
Sales / SDR & CoachOutbound prospecting and rep coaching.
Employee AgentInternal; runs as an authenticated employee inside Salesforce or Slack.
CustomAnything bespoke, assembled in Agent Builder.

The tell in a scenario is who the agent serves (customer versus employee) and whether there is an authenticated user.

Agent User & security

Every agent runs as an Agent User — a dedicated user record whose profile, permission sets and sharing rules define exactly what data and actions the agent can reach. The model: an agent can only ever do what its running user is permitted to do.

Trap

"The agent can't access X" or "a deployed action fails on permissions" is almost always a missing permission set assignment on the Agent User — or, for grounding, an inactive search index.

Deterministic control — filters & variables

Because the LLM is probabilistic, the exam tests how you make behaviour predictable. You use variables (state carried across the conversation) and filters / guard conditions (available_when-style logic gating whether a topic or action is eligible).

Trap

Declaring a variable and adding a guard condition is not enough. The variable must be mutable and explicitly updated at runtime (via an action or setVariables) for the guard to ever flip. A guard reading a variable nothing changes will never fire.

Channels

Agents surface through Messaging and digital experiences, email, Slack, and the Agent API for programmatic invocation. Match channel to type: a Service Agent on website messaging has no logged-in user; an Employee Agent in Slack does.

02  Domain · 20%

Prompt Engineering.

One workflow underpins the domain: Prompt Builder authors reusable prompt templates, you ground them with real Salesforce data so the model isn't inventing, and everything passes through the Einstein Trust Layer to and from the LLM.

Prompt template types

TemplateUse it when…
Sales EmailGenerating personalised email tied to a record (Contact, Lead, Opportunity).
Field GenerationPopulating or updating a specific record field with AI-generated text.
Record SummarySummarising a record and its related context for a user.
FlexGeneral purpose — free resource selection, multiple objects, custom logic. The catch-all.

The distinction that catches people: Field Generation writes into a field; Sales Email produces body copy; Flex is the answer when the requirement doesn't map to a purpose-built type.

Grounding — the heart of the domain

Grounding injects trusted data into the prompt so output is accurate, roughly in order of increasing power:

  • Merge fields — pull values directly from the record.
  • Related lists — bring in child records for context.
  • Flow and Apex — fetch or compute data the template can't reach declaratively (external systems, complex queries).
  • RAG via the Data Library / retrievers — ground on unstructured knowledge (the Domain 3 machinery).
Answer pattern

"The generated email keeps inventing product details" → ground it (add the missing data), not "rewrite the prompt to say don't make things up." Reducing hallucination is a grounding problem first, a wording problem second.

The Einstein Trust Layer

The single most-tested concept in the domain. Everything to and from the model passes through it.

Answer map

PII exposure → masking. "Not used to train the external model" → zero retention. Compliance / logging → audit trail.

Prompt best practices

  • Iterative & specific. Clear instructions, explicit constraints, and testing against real records in Prompt Builder's preview beat clever wording.
  • Few-shot vs chain-of-thought. Few-shot gives worked examples to imitate (good for format and consistency); chain-of-thought asks for step-by-step reasoning (good for multi-step logic and goal chaining — a commonly cited hard spot).

The permissions gotcha

Two permissions people conflate: managing templates (build and edit in Prompt Builder) versus executing them (a user running the template). "Can't see the template in Prompt Builder" is a manage gap; "the template won't run for end users" is an execute or access gap.

03  Domain · 20%

Data 360 for Agentforce.

The exam's angle here is narrow: it cares about the Agentforce Data Library as a grounding source for RAG, and the knobs that make retrieval accurate — not the broader Data 360 estate.

Data Library sources

SourceWhen it's the answer
Salesforce KnowledgeCurated Knowledge articles already exist; pick Identifying Fields to help the retriever locate the right article.
Files / documentsUnstructured content (PDFs, uploads) needing chunking and indexing.
Data 360 (DMOs)Grounding on data already unified in Data 360, via retrievers over data model objects.

Chunking & indexing

  • Chunk size — larger chunks capture more context but reduce precision; smaller chunks are precise but lose surrounding meaning. "Misses context spanning a paragraph" → increase chunk size or use semantic chunking.
  • Semantic chunking — splits on meaning rather than fixed length; the answer when intent or structure matters (e.g. technical content with ISO codes where naive splits break meaning).
  • Indexing — the Search Index must be active in the target org.
Trap

"Agent returns nothing after deployment" wearing Domain 3 clothing → inactive Search Index — sibling to the missing permission set from Domain 1.

Retrievers

  • Individual retriever — one index, one search configuration.
  • Ensemble retriever — combines multiple retrievers or sources and merges results; the answer when one library can't cover the breadth, or you need to blend structured and unstructured.

Search types — the highest-yield distinction

TypeMatches onPick it when…
KeywordExact lexical tokensCodes, SKUs, part numbers — the literal string matters.
VectorMeaning via embeddingsNatural-language questions; wording varies, intent is stable.
HybridBoth, fusedConceptual recall and exact-token precision — mixed content.

Classic setup: technical docs full of alphanumeric codes where semantic search keeps missing them → hybrid (or keyword weighting), not pure vector. Conversely, "customers phrase the same question ten ways" → vector.

Through-line

Retrieval still runs as the Agent User. A retriever can only surface records that user can see, so a data-sharing gap presents as "the agent can't find information that clearly exists." Trace it to permissions, not the index.

04  Domain · 20%

Deployment Lifecycle.

A fifth of the exam, and very learnable. Place any scenario on the lifecycle and the answer follows.

Test — the Agentforce Testing Center

Validate an agent at scale before it faces users. You feed a set of test utterances and it runs them through the agent in batch, checking for each: was the right topic selected, did the right action fire, was the response acceptable. This is where topic-routing and action-selection failures get caught systematically.

Answer pattern

"Validate agent behaviour across many scenarios before go-live" → Testing Center with batch utterances. Not manual testing in the preview panel.

Deploy — sandbox to production

Agent configuration is metadata, so it moves like any other Salesforce metadata: Change Sets (declarative, sandbox-to-related-production) and the Metadata API / SFDX (programmatic, CI/CD or unrelated orgs). Build in a sandbox, never straight in production.

The two go-live traps

"Works in sandbox, not prod" is almost always one of two things: the Agentforce permission set wasn't assigned to the relevant user in the target org, or the Search Index isn't active in production. Memorise the pair — they are the go-to distractors across the whole exam.

Monitor — adoption, quality, governance

  • Adoption — dashboards and reports on usage, session volume, deflection.
  • Quality — event logs and conversation analytics to trace what the agent actually did and spot reasoning failures in the wild.
  • CX metrics — be ready to read CSAT and NPS as adoption and quality signals.
  • Governance — who can build, review and release agents; monitoring for drift and misuse.
05  Domain · 5%

Multi-Agent Interoperability.

The theme: an agent rarely acts alone. It hands off to other agents, calls external tools, and can be invoked programmatically. The exam mostly tests which mechanism is which.

MechanismWhat it doesThe tell
MCPConnects an agent to external tools / data sources"connect to an external tool or third-party capability"
A2AAgent-to-agent delegation over Platform Events"one agent hands work to another agent"
Agent APIProgrammatic, external invocation of an agent"trigger the agent from an external application"

Beneath the mechanisms

  • Handoffs & escalation — a well-designed agent recognises it's out of scope and routes onward, to another agent or a human, rather than hallucinating past its limits.
  • Role clarity — when two agents could both handle something, define each agent's scope so ownership is unambiguous. It mirrors topic-scoping from Domain 1, one level up.
