01 · Objective

Declarative vs programmatic boundaries

Given a scenario, identify the boundaries of declarative customization and the use cases for programmatic customization.

This objective is the philosophical spine of the whole certification. Almost every scenario question is, underneath, asking the same thing: what is the simplest, most maintainable tool that fully satisfies this requirement? Get the instinct right and you will second-guess far fewer answers.

The build ladder

Think of platform customisation as a ladder you climb only as far as you must. Each rung adds power but also cost — more skill to build, more risk, more to maintain, harder to upgrade.

RungWhat lives hereCost of ownership
1 · ConfigurationStandard objects & fields, list views, standard reports — switch on and use.Lowest
2 · Declarative customisationCustom objects/fields, page layouts & Lightning pages, formula fields, validation rules, roll-ups, Flows, approval processes.Low — admin-maintainable
3 · ProgrammaticApex (triggers, classes, batch), Lightning Web Components, Aura, Visualforce, callouts.Highest — needs developers, tests, coverage
◆ The one rule

Choose the lowest rung that fully meets the requirement. On this exam, code is only the correct answer when the requirement is genuinely impossible — or clearly impractical — declaratively. If a Flow, formula, validation rule or roll-up can do it, an Apex answer is wrong even though it would also work.

When code is genuinely required

Learn the short list of things declarative tools cannot do, because the exam builds "you must use code" scenarios out of exactly these:

  • External callouts / integration logic too complex for a Flow HTTP action or External Services — Apex.
  • Complex, high-volume transactional logic — heavy bulk processing, custom rollback, multi-object orchestration beyond Flow's comfort.
  • Bespoke user interfaces that standard and AppExchange components can't render — Lightning Web Components.
  • Asynchronous / scheduled processing beyond scheduled Flows — Batch, Queueable, or @future Apex.
  • Logic that must run at points, or with control, Flow can't express — e.g. very complex cross-object validation or merge logic.
▲ Exam trap — the two directions

The trap runs both ways. Over-engineering: a validation-rule-shaped requirement with "write an Apex trigger" offered — wrong. Under-powering: a requirement that clearly needs a callout or complex bulk logic with "use a validation rule" offered — also wrong because it can't actually do the job. Read for what the requirement needs, then pick the lowest rung that reaches it.

Reading the signal words

Phrasing in the scenarioAlmost always points to
"without code", "point-and-click", "admins must maintain it"Declarative (Flow / formula / validation rule)
"prevent the record from saving when…"Validation rule
"automatically create/update a record when…"Record-triggered Flow
"calculate and display…"Formula field (or roll-up for children)
"call/sync an external system"Apex callout (or Flow HTTP action for simple cases)
"custom, pixel-perfect interface"Lightning Web Component
Worked scenario

A company needs a field on Case to show the parent Account's loyalty tier, and to stop users saving a Case if its Priority is High but no Account is linked. What should the admin build?

Reasoning → Answer

Two requirements, two declarative tools. Displaying a parent value is a cross-object formula field (Case → Account). Blocking a save on a condition is a validation rule. No code, no Flow needed. An answer proposing Apex, or a Flow to copy the tier onto the Case, over-engineers a formula problem.

◆ Chapter recap
  • Climb the ladder only as far as the requirement forces you: configuration → declarative → code.
  • Code is right only when declarative genuinely can't do it: callouts, complex bulk, bespoke UI, async.
  • Trap runs both ways — over-engineering and under-powering are both wrong answers.
  • Signal words ("without code", "prevent saving", "call an external system") map to tools.
Self-check.
Q1A requirement can be met with either a record-triggered Flow or an Apex trigger. Which does the exam prefer?
The Flow. — Lowest-code option that fully meets the requirement wins; Apex is only correct when declarative can't do the job.
Q2Name a requirement that forces you into Apex rather than Flow.
A complex external callout, heavy bulk/transactional logic, or bespoke UI (LWC). — These sit outside what declarative tools can express.
Q3Is "install an AppExchange app" a declarative or programmatic choice for the candidate?
It is a declarative, buy-don't-build choice. — You configure and install rather than write code, even though the package itself may contain Apex.
02 · Objective

Extending an org with the AppExchange

Identify common scenarios for extending an org using the AppExchange.

The AppExchange is Salesforce's marketplace — the "buy, don't build" rung. When a requirement describes a common, well-solved, non-differentiating need, the most professional answer is often to install someone else's solution rather than build your own.

What you can install

Listing typeWhat it is
AppsComplete solutions — document generation, e-signature, accounting, data quality.
ComponentsLightning components / building blocks that drop into your pages.
Flow & Bolt solutionsPre-built flows, actions, industry templates.
ConsultantsImplementation partners — services, not software.
◆ Delivery = packages

AppExchange solutions install as packages — almost always managed packages: namespaced, upgradeable by the publisher, and IP-protected. This ties directly into the App Deployment domain, so the two reinforce each other.

The build-vs-buy judgement

Buy from the AppExchange when the need is common and non-differentiating: mass email, deduplication, PDF/quote generation, complex billing, e-signature, address validation. Building these yourself is slower, costlier, and becomes your burden to maintain forever. Build custom only when the requirement is specific to your business and no vendor solves it well.

▲ Exam trap

When a scenario names a whole category that vendors clearly own — "generate a branded PDF quote", "clean and deduplicate a large import" — the best answer is usually "install an app from the AppExchange", not a bespoke build. The exam rewards not reinventing the wheel.

Before you install

  • Check the app's security review status, supported editions, and the limits it consumes (custom objects, fields, tabs).
  • Install into a sandbox first and test.
  • Choose install scope — admins only, all users, or specific profiles.
Worked scenario

Sales ops need reps to generate a branded, print-ready PDF proposal from an Opportunity, with no developer time available. What is the recommended approach?

Reasoning → Answer

Document generation is a mature, non-differentiating category with many managed apps on the AppExchange. Install a document-generation app rather than build Visualforce PDF rendering. It's faster, maintained by the vendor, and needs no code.

◆ Chapter recap
  • AppExchange is the buy-don't-build rung: apps, components, templates, and consultants.
  • Solutions install as packages — usually managed (upgradeable, namespaced, IP-protected).
  • Buy for common/non-differentiating needs; build only for what's specific to your business.
  • Check security review, editions, limits; install to a sandbox first.
Self-check.
Q1How do AppExchange solutions arrive in your org?
As packages, usually managed packages. — Which means the publisher can push upgrades and the source is IP-protected.
Q2A requirement is common across the industry and no developers are available. Build or buy?
Buy from the AppExchange. — Non-differentiating, well-solved needs are ideal buy candidates.
03 · Objective

Object, record & field access

Apply features and capabilities available to restrict and extend object, record, and field access.

Salesforce security is layered, and the exam tests whether you know which lever controls which layer. Confuse them and you'll pick the wrong tool for a security scenario every time.

The three layers

LayerQuestion it answersControlled by
Object (CRUD)Can the user Create/Read/Update/Delete this type of record at all?Profiles & Permission Sets
Field (FLS)Can the user see/edit this specific field?Profiles & Permission Sets (Field-Level Security)
RecordWhich specific records can the user see/edit?OWD, role hierarchy, sharing rules, manual sharing, teams, Apex

This chapter covers the top two (object and field). Record-level sharing is big enough to have its own chapter next.

Profiles vs permission sets

  • Every user has exactly one profile — the baseline. A profile grants object CRUD, field FLS, tab visibility, app access, page-layout assignment, login hours/IP, and system/user permissions.
  • Permission sets and permission set groups grant additional access on top of the profile. They only add.
◆ The modern best-practice model

Keep profiles lean and generic (a small number of baseline profiles) and layer every exception through permission sets. When a scenario says "a subset of users also needs access to X", the answer is a permission set, not a brand-new profile.

▲ Exam trap — add vs restrict

A permission set can only extend access; it can never take away what the profile grants. To restrict, you tighten the profile. (Within a permission set group, a muting permission set can remove specific permissions — but a standalone permission set never subtracts.)

How CRUD and FLS interact

Field access is meaningful only when object access exists. To see a field's data a user needs Read on the object AND Visible on the field. To edit it, they need Edit on the object AND the field not marked Read-Only in FLS. Removing object Read hides every field regardless of FLS.

Worked scenario

Most users should never see the Contact "SSN" field, but a small compliance team must edit it. How do you configure this?

Reasoning → Answer

Set FLS on the baseline profile(s) so the field is not visible to everyone. Then grant the compliance team a permission set that makes the field visible and editable, and assign it only to them. Extend for the few; restrict at the baseline.

◆ Chapter recap
  • Three layers: object (CRUD) and field (FLS) via profiles + permission sets; record via sharing.
  • One profile per user = baseline; permission sets add on top and only ever extend.
  • "A subset also needs X" → permission set. To restrict → tighten the profile.
  • Seeing a field needs Read on the object AND Visible on the field.
Self-check.
Q1A group of users needs extra object access beyond their profile. New profile or permission set?
Permission set. — Keep profiles lean; layer exceptions with permission sets.
Q2Can a (standalone) permission set remove access the profile grants?
No. — Permission sets only extend. Restricting is done on the profile; only a muting permission set within a group can remove.
Q3A user has Read on the object but the field is hidden in FLS. Do they see the field?
No. — Both object Read and field Visible are required.
04 · Objective

Choosing the right sharing solution

Given a set of business requirements, determine the appropriate sharing solution.

Record-level access is a system unto itself. It starts locked at the org-wide default and is progressively opened up. The golden principle: sharing always widens access, it never restricts below the baseline.

Org-Wide Defaults — the floor

OWDMeaning
PrivateOnly the owner (and those above in the role hierarchy) can see the record.
Public Read OnlyEveryone can view; only owner/above can edit.
Public Read/WriteEveryone can view and edit.
Controlled by ParentAccess mirrors the master record (master-detail children).

Set the OWD to the most restrictive level anyone should have, then open up for those who need more.

The mechanisms that widen access

  1. Role hierarchy — managers automatically gain access to records owned by people beneath them (when "Grant Access Using Hierarchies" is enabled; it's always on for standard objects).
  2. Sharing rules — automatic rules based on record owner or criteria, sharing whole categories of records to roles, subordinates, or public groups. Grant Read Only or Read/Write.
  3. Manual sharing — an owner or admin shares an individual record with a specific user/group (only available when OWD is Private or Public Read Only).
  4. Teams — Account, Opportunity, and Case teams grant access via membership.
  5. Apex managed sharing — programmatic, for logic too complex for declarative rules.
◆ Mnemonic — O-R-S-M-A

OWD → Role hierarchy → Sharing rules → Manual → Apex. The order in which access widens, from broad baseline to surgical exception.

Choosing between them

Requirement shapeSolution
Everyone should have the same accessSet the OWD
Managers should see their reports' recordsRole hierarchy (usually automatic)
A whole role/group needs a category of recordsSharing rule (criteria- or owner-based)
Share one specific record with one personManual sharing
Collaboration on specific accounts/opps/casesTeams
Logic too complex for rulesApex managed sharing
▲ Exam trap

If OWD is Public Read/Write, sharing rules and manual sharing are pointless — access is already maximal. They only matter when the OWD is more restrictive than everyone-can-edit. Also: sharing can raise Read to Read/Write but can never drop anyone below the OWD.

Worked scenario

Opportunities are Private. The EMEA sales team (a role) must all read every Opportunity owned by anyone in the EMEA region. Occasionally a rep needs to grant one specific competitor-sensitive Opp to a single legal reviewer. What two mechanisms fit?

Reasoning → Answer

The team-wide, category-based need is a sharing rule (share EMEA-owned Opps to the EMEA role, Read Only). The one-off, single-record, single-person need is manual sharing. Two different shapes, two different tools — both legitimate only because OWD is Private.

◆ Chapter recap
  • OWD sets the most restrictive floor; every other mechanism only opens access up.
  • O-R-S-M-A: OWD, role hierarchy, sharing rules, manual, Apex.
  • Group needs a category → sharing rule; one record to one person → manual sharing.
  • With Public Read/Write OWD, sharing rules and manual sharing are meaningless.
Self-check.
Q1A public group needs read access to all cases created by the support team. Which mechanism?
A sharing rule (owner- or criteria-based). — It shares a whole category of records to a group automatically.
Q2Does a sharing rule work if OWD is Public Read/Write?
No — it's redundant. — Everyone already has maximal access; sharing only widens from a restrictive baseline.
Q3Requirement: share exactly one record with exactly one user, ad hoc. Tool?
Manual sharing. — Available when OWD is Private or Public Read Only.
05 · Objective

Reports — formats and analysis

Identify the features and capabilities available when creating reports.

Report questions test two things: choosing the right format for the shape of the analysis, and knowing which analysis features exist so you don't reach for a custom field or code when the report engine already does it.

The four formats

FormatUse whenCharts / dashboards?
TabularA simple list of rows, no grouping. Fastest; good for exports and list-style output.Only on a dashboard with a row limit
SummaryRows grouped with subtotals (Opps by Stage). The workhorse.Yes
MatrixGrouped by rows and columns — a pivot grid (Stage × Month).Yes
JoinedMultiple blocks side by side, possibly different report types, sharing a common field.Yes
▲ Exam trap

Charts and most dashboard components need grouped data — so Summary or Matrix. A Tabular report can only feed a dashboard component if you set a row limit. "We need a chart but the report is tabular" → change it to Summary.

Analysis features to recognise

  • Bucket fields — categorise on the fly (deal-size bands) without creating a custom formula field.
  • Summary formulas — calculated columns at row or group level (win rate); PARENTGROUPVAL / PREVGROUPVAL compare across groupings.
  • Row-level formulas — a calculation per row, again without a stored field.
  • Cross filters — filter on the presence/absence of related records ("Accounts with/without Opportunities").
  • Filter logic — AND/OR combinations across filters.
  • Historical trend reporting & reporting snapshots — track how values change over time.
◆ Report before field

When a scenario needs a one-off categorisation or calculation for a report only, prefer a bucket field or report/row-level formula over creating a permanent custom field. Less clutter, no data model change.

Worked scenario

A manager wants a report showing which Accounts have no open Cases, grouped by Region, with a chart. Which format and features?

Reasoning → Answer

Summary format (grouped by Region, so a chart is possible). Use a cross filter: Accounts without Cases (with a sub-filter on Status = Open). No custom field, no code.

◆ Chapter recap
  • Four formats: Tabular (list), Summary (grouped), Matrix (pivot), Joined (blocks).
  • Charts/dashboards need grouping → Summary or Matrix; Tabular needs a row limit.
  • Buckets, summary/row-level formulas, cross filters and filter logic solve most needs without custom fields.
  • Prefer report-only features over permanent custom fields for one-off analysis.
Self-check.
Q1You need a two-dimensional pivot of revenue by Product (rows) and Quarter (columns). Format?
Matrix. — Grouping on both rows and columns.
Q2How do you report "Accounts with no Opportunities"?
A cross filter (Accounts without Opportunities). — Cross filters test presence/absence of related records.
Q3A tabular report needs to appear as a dashboard chart. What must you do?
Set a row limit (or convert to Summary). — Charts require grouped data or a limited tabular source.
06 · Objective

Report types

Identify the features and capabilities available when creating report types.

A report type is the lens that decides which objects and fields a report can even draw from — and, crucially, whether related child records are required. Choose the wrong report type and the data you want is simply unreachable, no matter how you build the report.

Standard vs custom report types

  • Standard report types are auto-generated for standard/custom objects and their relationships.
  • Custom report types let you define exactly which related objects appear, which fields are available (including fields pulled via lookup), and the join behaviour.
◆ The "with" vs "with or without" switch

In a custom report type each object relationship is set to one of:

  • "A records must have related B records" — an inner join. Only Accounts that have Cases appear.
  • "A records may or may not have related B records" — an outer join. All Accounts appear, with or without Cases.
▲ Exam trap

"Our report is missing records that have no child records" → the report type is set to "must have related"; switch it to "may or may not have". This single setting is one of the most reliably tested facts in the whole domain. A custom report type can join up to four objects.

Other capabilities

  • Surface fields from related objects by adding fields via lookup.
  • A custom report type moves from In Development to Deployed before users can build on it.
  • Control which fields appear by default in the report builder layout.
Worked scenario

A custom object "Project" has related "Milestones". A report on the standard type shows only Projects that already have Milestones, but the PMO needs to see all Projects including empty ones. What do you change?

Reasoning → Answer

Build (or edit) a custom report type and set the Project→Milestone relationship to "Projects may or may not have related Milestones" (outer join). Now empty Projects appear too.

◆ Chapter recap
  • A report type defines which objects and fields a report can access.
  • Custom report types add control over relationships and lookup fields; join up to 4 objects.
  • "Must have related" = inner join; "may or may not have" = outer join.
  • Missing childless records → switch the relationship to "may or may not have".
Self-check.
Q1A report omits parent records that have no children. What's the fix?
Set the custom report type relationship to "may or may not have related". — That's the outer join; the inner join hides childless parents.
Q2How many objects can a custom report type join?
Up to four. — Beyond four you need a different approach.
07 · Objective

Dashboards

Identify the features and capabilities available when creating dashboards.

A dashboard is a visual layer over report data. The single most tested dashboard concept is the running user — whose data everyone sees.

Fundamentals

  • Each component is powered by a source report (chart, gauge, metric, table, etc.).
  • A dashboard shows a snapshot, refreshed on demand or on a schedule — not live data.

The running user

◆ Running user

A dashboard displays data as seen by its running user, honouring that user's sharing access. Everyone who views a standard dashboard sees the same data — the running user's.

TypeRunning userUse case
StandardA fixed, single running userEveryone sees one shared view (e.g. the VP's org-wide numbers)
Dynamic"The logged-in user"Each viewer sees their own data through one dashboard
▲ Exam trap

"Each rep should see only their own numbers, from one dashboard" → dynamic dashboard (runs as the logged-in user). "Everyone should see the same regional roll-up" → standard dashboard with that manager as running user. Dynamic dashboards have per-edition limits (e.g. up to 10 in Enterprise) and can't be scheduled for refresh or used for following.

Filters and access

  • Dashboard filters let a viewer slice the same dashboard (up to 3 filters).
  • Access is governed by the dashboard folder, shared to users, roles, or public groups.
Worked scenario

Leadership wants one sales dashboard that every rep can open, where each rep sees only their own pipeline — without building 40 dashboards. Solution?

Reasoning → Answer

A single dynamic dashboard set to run as the logged-in user. Each rep sees their own data through the one dashboard, respecting their record access.

◆ Chapter recap
  • Dashboards show a snapshot of report data through the running user's access.
  • Standard = fixed running user (everyone sees the same); dynamic = logged-in user (each sees own).
  • Dynamic dashboards have edition limits and can't be scheduled.
  • Folder sharing controls who can view; up to 3 dashboard filters.
Self-check.
Q1Whose data does a standard dashboard show?
The running user's data, for everyone who views it. — It honours the running user's sharing, not the viewer's.
Q2One dashboard, each viewer sees their own records. Which type?
Dynamic dashboard (run as logged-in user). — Avoids cloning a dashboard per person.
08 · Objective

Actions & layouts for mobile

Given a set of requirements, determine the appropriate global, object-specific actions and layouts to optimise the Salesforce mobile user experience.

The Salesforce mobile app renders the same metadata as desktop, but two levers decide whether it's usable on a phone: actions (what users can quickly do) and compact layouts (what they see first).

Global vs object-specific actions

TypeWhere it livesKey trait
Global actionAvailable anywhere — mobile nav, publisher, Home. Not tied to a record.No parent context
Object-specific actionOn a specific object's page layout.Can auto-relate and pre-fill fields from the parent record

Action categories include Create a Record, Update a Record, Log a Call, Send Email, and Custom (Flow / LWC / Visualforce).

◆ Why object-specific wins for context

Object-specific actions can set the relationship automatically and pre-populate fields from the parent — a global action can't, because it has no parent. "New Case" launched from an Account, pre-filling that Account, is object-specific.

Compact layouts — the mobile headline

The compact layout defines the key fields shown at the top of a record in the mobile app, and in the highlights panel on desktop Lightning. It's how you decide what a user sees first on a small screen.

▲ Exam trap

"Reps need the four most important fields visible immediately on their phone" → configure the compact layout, not the full page layout. The full page layout still governs the complete detail view and which actions appear in the mobile action bar.

Also relevant

  • The page layout's mobile & Lightning actions section orders actions in the mobile action bar.
  • The Salesforce mobile app navigation menu is configured per app/profile.
Worked scenario

Field reps complain they must scroll through a long Account layout on mobile to find the phone number and tier. They also want to log a call in two taps, defaulting the related Account. What do you configure?

Reasoning → Answer

Set the Account compact layout to surface phone and tier at the top. Add an object-specific "Log a Call" action on the Account layout so it auto-relates to that Account. Compact layout for what they see; object-specific action for what they do.

◆ Chapter recap
  • Global actions work anywhere but have no parent context; object-specific actions pre-fill from the parent.
  • Compact layout = the key fields shown first on mobile and in the desktop highlights panel.
  • Full page layout still drives the detail view and the action bar order.
  • "See key fields first" → compact layout; "quick action in context" → object-specific action.
Self-check.
Q1You need the four most important fields at the top of a record on mobile. What do you edit?
The compact layout. — It controls the highlighted fields on mobile and the desktop highlights panel.
Q2Which action type can pre-populate fields from the parent record?
Object-specific. — Global actions have no parent context.
09 · Objective

Chatter — collaboration

Describe the customisations and use cases for Chatter.

Chatter is the platform's collaboration layer — feeds, groups, and feed tracking that turn records into conversations. The exam's favourite Chatter fact is feed tracking, and the distinction between it and field history tracking.

What Chatter provides

  • Feeds on records and profiles — posts, comments, @mentions, likes, files.
  • Groups — public, private, or unlisted collaboration spaces, optionally tied to records.
  • Topics & hashtags — cross-cutting organisation of posts.
  • Chatter actions in the publisher — the same global/object-specific actions surface here.

Feed tracking — the key control

◆ Feed tracking

Enable feed tracking on an object and select up to 20 fields; changes to those fields post automatically to the record's feed, visible to followers. It's the declarative way to surface "what changed" in context.

▲ Exam trap — don't confuse these

Chatter feed tracking posts changes into the feed for collaboration. Field history tracking is a separate feature that stores an audit trail (up to 20 fields) for reporting, not shown in the feed. "Users should see key changes in context on the record" → feed tracking. "We need an auditable history for compliance reporting" → field history tracking.

Common use cases

  • Cross-functional deal collaboration on the Opportunity feed.
  • Groups for projects, regions, or communities of practice.
  • Broadcasting record changes and approvals to stakeholders.
  • Knowledge sharing via posts, files, and topics.
Worked scenario

Support wants agents to see, right on the Case, whenever Priority or Status changes — as a running conversation they can comment on. What do you enable?

Reasoning → Answer

Enable Chatter feed tracking on the Case object for the Priority and Status fields. Changes post to the Case feed where agents can comment. Field history tracking would log the changes for reports but wouldn't put them in the collaborative feed.

◆ Chapter recap
  • Chatter = feeds, groups, topics, and publisher actions for collaboration.
  • Feed tracking (up to 20 fields) posts changes into the record feed.
  • Field history tracking is a separate, report-only audit trail — don't confuse the two.
  • Use groups and feeds for cross-functional collaboration and change visibility.
Self-check.
Q1You want field changes to appear in the record's Chatter feed. What do you enable?
Feed tracking on those fields. — Up to 20 fields per object.
Q2Compliance needs an auditable, reportable history of a field. Feed tracking or field history tracking?
Field history tracking. — It stores a reportable audit trail rather than posting to the feed.
