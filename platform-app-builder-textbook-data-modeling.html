Data Modeling & Management.

How you shape the data model silently decides sharing, roll-ups, cascade delete, reporting, and required-ness. This is the most consequential set of design choices on the platform — and the exam tests the side-effects, not just the definitions.

01 · Objective

Determining the appropriate data model

Given a scenario, determine the appropriate data model — standard objects, custom objects, and record types.

Good modelling starts by using what the platform already gives you. Reach for a custom object only when no standard object is a sensible home for the data, and reach for a record type when one object must behave differently for different segments.

Standard objects and their relationships

ObjectRepresentsKey relationships
AccountA company/organisation (or a person via Person Accounts)Parent to Contacts, Opportunities, Cases
ContactAn individual, usually under an AccountChild of Account; Contact Roles on Opps/Cases
LeadAn unqualified prospect (person + company combined)Converts to Account + Contact + Opportunity
OpportunityA potential dealChild of Account; has line items (Products) and Stages
CaseA service issueChild of Account/Contact
CampaignA marketing initiativeMany-to-many with Leads/Contacts via Campaign Members
▲ Exam trap — lead conversion

Converting a Lead creates an Account, a Contact, and (optionally) an Opportunity. Custom lead fields do not carry across automatically — you must configure lead field mapping, or that data is lost on convert.

Record types — one object, many processes

◆ What a record type controls

A record type lets a single object serve different business processes with different picklist values, different page layouts (assigned per record type per profile), and different business processes (sales, support, lead processes). Assignment is controlled per profile and permission set.

Use record types when the same kind of record behaves differently for different segments — one Opportunity object with "New Business" and "Renewal" record types, each with its own stages and layout, is the canonical example.

▲ Exam trap — what record types do NOT do

Record types do not control field-level security or record access. They only vary picklists, layouts, and process. "Different groups need different fields hidden" is FLS, not record types. "Different groups need different picklist values and layouts" is record types.

Custom objects and external objects

  • Custom objects (__c) — when no standard object fits. They get the full toolkit: fields, relationships, layouts, reports, automation.
  • External objects (__x) — map to data stored outside Salesforce via Salesforce Connect. Data is fetched live, never stored — ideal for very large external datasets you shouldn't copy in.
Worked scenario

A company sells to both businesses and consumers. Opportunities for each need different stages, different fields on the layout, and a different sales process — but they should report together as Opportunities. Model?

Reasoning → Answer

One Opportunity object with two record types ("B2B", "B2C"), each mapped to its own sales process (stage set) and page layout, assigned by profile. A separate custom object would fragment reporting; FLS wouldn't change stages or layouts.

◆ Chapter recap
  • Use standard objects first; custom objects only when nothing fits; external objects for live external data.
  • Record types vary picklists, layouts, and business processes — assigned per profile.
  • Record types do NOT control FLS or record access.
  • Lead conversion creates Account + Contact + (optional) Opportunity; map custom fields or lose them.
Self-check.
Q1One object needs two different sets of stages and layouts for two segments. Tool?
Record types (each mapped to a business process and layout). — Assigned per profile; keeps reporting unified.
Q2Do record types restrict which fields a user can see?
No — that's field-level security. — Record types only affect picklists, layouts, and process.
Q3You must display a large external dataset live without importing it. What object type?
External object via Salesforce Connect. — Data is fetched on demand, not stored.
02 · Objective

Relationship types and their implications

Given a scenario, explain the capabilities of the various relationship types and the implications of each on record access, user interface, and reporting.

The relationship you choose is never just a link — it silently sets sharing, roll-ups, cascade delete, ownership, required-ness, and reporting behaviour. This is the highest-value chapter in the domain.

Master-Detail vs Lookup

Master-DetailLookup
CouplingTight — child depends on parentLoose — independent
Parent required?Always requiredOptional (can be made required)
Delete behaviourCascade delete — delete parent, children go tooParent normally can't be deleted while referenced; optional clear/prevent
Record ownershipChild has no owner — inherits the parent'sChild has its own owner
Sharing / securityChild access controlled by parentIndependent sharing
Roll-up summary on parent?YesNo (a few standard exceptions)
ReparentingNot allowed by default (option to allow)Freely reparented
Limit per objectUp to 2 master-detail fieldsUp to 40 relationships total
◆ The deciding questions

Need a roll-up summary on the parent? → master-detail (the only native way). Should deleting the parent delete the children? → master-detail. Should the child stand alone, be optional, or have its own sharing? → lookup.

◆ Mnemonic — ROCS

Master-detail gives you Roll-ups, Ownership inherited, Cascade delete, Sharing controlled by parent. If the scenario wants any of ROCS, it wants master-detail.

Many-to-many via a junction object

Salesforce has no native many-to-many field. You build a junction object with two master-detail relationships, one to each parent.

  • The first master-detail defined is the primary: it governs the junction record's ownership, sharing, look-and-feel (colour, layout), and its cascade behaviour.
  • The second is the secondary parent.
  • Classic example: Job ↔ Candidate via an "Application" junction; or the standard Campaign Member bridge.
▲ Exam trap

"A Position can have many Candidates and a Candidate many Positions" → junction object with two master-detail relationships. If the junction uses lookups instead, you lose cascade delete and roll-up capability. And if you later need to delete a parent freely, remember master-detail cascades to the junction records.

The specialised relationships

  • Hierarchical — a special lookup available only on the User object (e.g. "Manager").
  • External lookup — relates a child to an external object via the parent's External ID.
  • Indirect lookup — relates an external object child to a standard/custom parent by matching an External ID field.
  • Self-relationship — a lookup from an object to itself (Account → Parent Account).
Worked scenario

Each Invoice has many Line Items. The business needs the Invoice to show a Total (sum of line amounts), and deleting an Invoice must remove its Line Items. Line Items should never exist without an Invoice. Relationship?

Reasoning → Answer

Master-detail (Line Item → Invoice). It satisfies the roll-up (Total), the cascade delete, and the "can't exist alone" requirement (parent required). A lookup would allow orphan Line Items and couldn't roll up the Total natively.

◆ Chapter recap
  • Master-detail = ROCS: Roll-ups, Ownership inherited, Cascade delete, Sharing by parent; parent required; max 2/object.
  • Lookup = loose, optional, own owner/sharing, no roll-up, freely reparented.
  • Many-to-many = junction object with two master-details; the first is primary (owner, sharing, look/feel).
  • Hierarchical (User only), external/indirect lookup (external objects), self-relationship round out the set.
Self-check.
Q1You need to roll up child amounts onto the parent. Which relationship is required?
Master-detail. — Roll-up summaries need a master-detail relationship (barring a few standard cases).
Q2How do you model a many-to-many between Course and Student?
A junction object (e.g. Enrolment) with two master-detail relationships. — The first master-detail defined controls ownership, sharing, and look-and-feel.
Q3A child record must be able to exist independently and have its own owner. Master-detail or lookup?
Lookup. — Master-detail children inherit ownership and require a parent.
03 · Objective

Selecting & changing field data types

Given a scenario, determine the considerations when selecting or changing a field data type.

Two sub-skills: picking the right field type up front, and knowing which type changes quietly destroy existing data. The exam is fond of "what happens to the data if we convert field X?"

Choosing the right type

NeedField type
Single choice from a fixed setPicklist
Several choices from a setMulti-select picklist — use sparingly (poor for reporting/filtering)
Calculated, read-only valueFormula
Auto-incrementing unique identifierAuto Number
MoneyCurrency
Yes / noCheckbox
Link to another recordLookup / Master-Detail
Unique key for integration / matchingText or Number with External ID + Unique
◆ External ID

Flagging a field as External ID indexes it and makes it usable as the match key for upserts and integrations. When a scenario mentions matching Salesforce records to an external system's key, an External ID field is usually part of the answer.

Changing a field's type — the data-loss rules

▲ The big one

Several conversions truncate or delete existing data. The safe procedure is always: export the data, test the change in a sandbox, then convert in production.

ConversionEffect on data
Text → PicklistBroadly workable, but values not in the picklist may need cleanup; a restricted picklist can reject/strip values
Picklist → TextSafe — values become text
Text ↔ Text AreaSafe
Long Text Area → Text / Text AreaTruncates everything beyond the shorter limit — data loss
Number / Currency / Percent (interchange)Usually safe; watch decimal places and length
To Checkbox, Date, Date/Time, Number, Currency, Percent (from text)Salesforce warns of data loss; values that don't fit the new type are cleared
Auto NumberChanging to Auto Number reassigns values; changing from Auto Number keeps the existing text
◆ Change restrictions
  • Can't change a field referenced by Apex, a formula, or a roll-up without resolving the dependency first.
  • Can't convert to a type that breaks an existing validation rule or field update.
  • Some conversions are simply not permitted.
Worked scenario

A "Notes" field is a Long Text Area (up to 32,000 chars) but the team wants it as a Text(255) so it fits a column. Several records hold 2,000+ characters. What happens, and what's the safe path?

Reasoning → Answer

Converting Long Text Area → Text(255) truncates everything past 255 characters — permanent data loss. Safe path: export the field first, test the conversion in a sandbox, confirm the impact, and only then decide. Often the right answer is don't shorten it.

◆ Chapter recap
  • Pick types by purpose; External ID enables upsert/integration matching.
  • Long Text Area → Text truncates; converting to Checkbox/Number/Currency/Date can clear non-fitting data.
  • Always export + sandbox-test before changing a field type in production.
  • Fields referenced by Apex/formulas/roll-ups or breaking validation rules can't be freely converted.
Self-check.
Q1What happens converting a Long Text Area to Text(255) when records exceed 255 chars?
Data beyond 255 characters is truncated (lost). — Test in a sandbox and export first.
Q2Which field attribute lets you upsert records matched to an external system?
External ID (ideally + Unique). — It indexes the field and enables upsert matching.
Q3Can you change the type of a field referenced in a validation rule freely?
No — not if the change breaks the rule. — Resolve dependencies first.
04 · Objective

Schema Builder

Explain the capabilities and considerations of Schema Builder.

Schema Builder is the visual, drag-and-drop view of the data model. It's excellent for seeing and sketching, but it deliberately stops short of detailed configuration — and the exam tests exactly where it stops.

What it can do

  • Visualise the whole model — objects and relationships as an entity diagram.
  • Create custom objects and drag on new fields and relationships directly.
  • Show/hide objects, view the legend, auto-arrange the layout.
◆ Best for

Quickly understanding an unfamiliar org's model, and rapidly roughing out new objects, fields, and relationships without stepping through the setup wizards.

What it cannot do — the considerations

▲ Exam trap

Schema Builder cannot:

  • Set field-level security or add fields to page layouts — fields created here aren't auto-added to layouts.
  • Configure validation rules, record types, or most field attributes beyond creation.
  • Delete objects or fields.

So detailed configuration still happens in the standard Object Manager.

Worked scenario

An admin uses Schema Builder to add three fields to a custom object, then can't find them on the record page. Why?

Reasoning → Answer

Fields created in Schema Builder are not automatically added to page layouts, and Schema Builder doesn't set FLS. The admin must add them to the page layout (and check field-level security) in Object Manager.

◆ Chapter recap
  • Schema Builder visualises the model and creates objects, fields, and relationships by drag-and-drop.
  • It does NOT set FLS, add fields to layouts, build validation rules, or delete components.
  • Use it for design and light building; finish configuration in Object Manager.
Self-check.
Q1Can you set field-level security in Schema Builder?
No. — FLS and layout placement happen in Object Manager.
Q2A new field made in Schema Builder isn't on the layout. Expected?
Yes — Schema Builder doesn't auto-add fields to page layouts. — Add it manually afterwards.
05 · Objective

Importing, exporting & external data

Given a scenario, determine the options and considerations when importing and exporting data, including the capabilities of external data sources.

Match the tool to the volume, the object, and the operation (insert / update / upsert). Then decide whether the data should be imported at all, or accessed live via Salesforce Connect.

Import tools

ToolVolumeNotes
Data Import WizardUp to 50,000 recordsIn-browser. Accounts, Contacts, Leads, Campaign Members, custom objects. Can optionally trigger workflows/flows. No delete.
Data LoaderUp to 5 million recordsDesktop app / CLI. All objects. Insert, update, upsert, delete, export. For large or complex loads.
◆ Wizard vs Loader shortcut

≤50k, standard objects, admin in-browser → Data Import Wizard. >50k, any object, need upsert/delete/scheduling → Data Loader.

Insert / update / upsert

  • Insert — create new records.
  • Update — change existing records, matched by Salesforce Id.
  • Upsert — insert or update in one pass, matched by an External ID field. The go-to for syncing from an external system without duplicates.
▲ Exam trap

"Load records from another system — update existing, create new, no duplicates" → upsert on an External ID. Plain insert duplicates; plain update skips new records.

Considerations before importing

  • Deduplicate with matching and duplicate rules.
  • Map fields carefully — picklists, record types, owners.
  • Decide deliberately whether the load should fire or bypass automation.
  • Respect storage and field-type constraints.

External data — Salesforce Connect

When you need to see external data live without importing it, use Salesforce Connect with external objects (OData 2.0/4.0 or a custom Apex adapter). Data stays in the source system and is fetched on demand — ideal for very large datasets or a system of record you must not copy.

Worked scenario

A nightly job must sync 800,000 product records from an ERP into Salesforce, updating changed ones and adding new ones without creating duplicates. Tool and operation?

Reasoning → Answer

Data Loader (volume far exceeds the Wizard's 50k), running an upsert matched on an External ID (the ERP's product key). If the products shouldn't live in Salesforce at all, Salesforce Connect with external objects is the alternative.

◆ Chapter recap
  • Data Import Wizard: ≤50k, in-browser, common objects, no delete.
  • Data Loader: ≤5M, all objects, insert/update/upsert/delete/export.
  • Upsert on an External ID = update-or-create without duplicates.
  • Salesforce Connect / external objects surface external data live without importing it.
Self-check.
Q170,000 custom-object records need loading with upsert. Wizard or Loader?
Data Loader. — Volume exceeds the Wizard's 50k limit and the Wizard can't upsert to all objects the same way.
Q2What operation avoids duplicates when syncing from an external system?
Upsert on an External ID field. — It matches existing records and inserts new ones in one pass.
Q3You must show millions of external records live without storing them. Approach?
Salesforce Connect with external objects. — Data is fetched on demand, not imported.
