The smallest domain, but rich in crisp, memorisable facts: the four sandbox types and their refresh intervals, what change sets can and can't do, managed vs unmanaged packages, and the deployment mechanism for each situation.

01 · Objective

Application lifecycle & sandbox types

Given a set of business requirements, recommend a solution for key milestones and considerations when managing the application lifecycle and various types of sandboxes.

The healthy lifecycle keeps changes out of production until tested. You build and test in sandboxes, then promote to production. The four sandbox types differ in the data they copy and how often they refresh — both are frequently tested.

The lifecycle

Develop → test → deploy. Use sandboxes for build, QA, UAT, and staging; deploy tested changes to production. Never build directly in production where it can be avoided.

The four sandbox types

SandboxData copiedRefresh intervalBest for
DeveloperMetadata only (config, no records)1 dayIndividual development / basic testing
Developer ProMetadata only; larger storage1 dayDevelopment with more test data you load
Partial CopyMetadata + a sample of data (sandbox template, ~10k records/object)5 daysQA / UAT with representative data
FullMetadata + all production data29 daysStaging, performance/load, full UAT
◆ The two everyone confuses

Partial Copy = a sample of data (template-defined, capped per object), refresh every 5 days. Full = a complete copy of production, refresh every 29 days. Developer and Developer Pro carry no records — only configuration.

▲ Exam trap

"Realistic, representative dataset but not the whole database" → Partial Copy. "Production-scale data for performance testing / final staging" → Full. "Just configuration, no data" → Developer. The refresh intervals (1 / 1 / 5 / 29 days) are a favourite standalone question.

Worked scenario

Before a major release, the team needs a final staging environment that mirrors production data volumes to test performance and run full UAT. Which sandbox?

Reasoning → Answer

A Full sandbox — it's the only type with a complete copy of production data, suited to performance/load testing and final staging. It refreshes every 29 days, which fits an infrequent pre-release cadence.

◆ Chapter recap
  • Build/test in sandboxes, then deploy to production.
  • Developer & Developer Pro = metadata only, 1-day refresh; Developer Pro has more storage.
  • Partial Copy = sample data (~10k/object), 5-day refresh; Full = all data, 29-day refresh.
  • Match the sandbox to the data realism the task needs.
Self-check.
Q1Which sandbox carries a complete copy of production data?
Full sandbox. — Refreshes every 29 days; used for performance testing and staging.
Q2Refresh interval of a Partial Copy sandbox?
5 days. — It holds a sample of data (~10k records per object).
Q3Which sandboxes contain no records at all?
Developer and Developer Pro. — They copy metadata only.
02 · Objective

Change sets

Given a use case, demonstrate knowledge, viability, and troubleshooting when using change sets.

Change sets are the declarative, admin-friendly way to move metadata between related orgs. They're simple and useful — but their limits are exactly what the exam probes.

How they work

  • An outbound change set is created in the source org (usually a sandbox), populated with components, and uploaded.
  • It arrives as an inbound change set in the target org (usually production), where it's validated and deployed.
  • They require a deployment connection and travel only between related orgs — a production org and its sandboxes.
◆ Add dependencies

Use "View/Add Dependencies" to pull in components a change relies on (a field's object, a validation rule's fields). A missing dependency is the number-one cause of a failed deployment.

▲ Exam trap — what change sets can't do
  • Deploy to unrelated orgs — only between a production org and its own sandboxes.
  • Delete components in the target — you can only create/update.
  • Move data — metadata only.

For unrelated orgs or deletions, use packages or the Metadata API / SFDX / ANT.

Before the real deployment, run a validation-only deploy on the inbound change set to catch failures (missing dependencies, failing tests) early.

Worked scenario

An admin deploys a new validation rule via change set, but it fails because it references two fields not present in production. What went wrong and how is it avoided next time?

Reasoning → Answer

The change set was missing dependencies — the two referenced fields. Use View/Add Dependencies to include them, and run a validation-only deploy first to surface the gap before the live deployment.

◆ Chapter recap
  • Change sets move metadata between related orgs (production and its sandboxes) via a deployment connection.
  • They can't reach unrelated orgs, can't delete components, and can't move data.
  • Add dependencies to avoid the most common failure; validate before deploying.
  • For unrelated orgs or deletions, use packages or the Metadata API.
Self-check.
Q1Can a change set delete a component in production?
No. — Change sets only create/update; use the Metadata API (destructiveChanges) to delete.
Q2Change sets can move metadata between which orgs?
Related orgs only — a production org and its sandboxes. — Unrelated orgs need packages or the Metadata API.
Q3Top cause of a failed change-set deployment?
Missing dependencies. — Use View/Add Dependencies and validate first.
03 · Objective

Managed & unmanaged packages

Describe the use cases and considerations when using unmanaged and managed packages.

Packages bundle metadata for distribution. The managed-vs-unmanaged choice turns on two things: upgradeability and IP protection.

The comparison

UnmanagedManaged
PurposeDistribute editable templates / open source; move metadataDistribute a product (ISV / AppExchange)
Editable after install?Yes — becomes the subscriber's ownMostly locked; some components extensible
Upgradeable by publisher?NoYes — versioned upgrades
NamespaceNoneNamespaced (isolated)
IP protected?No (source visible)Yes (source hidden)
◆ Choose by intent

Distributing a maintained, upgradeable, protected app to many customers → managed. Sharing an editable starting point / open-source template, or moving metadata between your own orgs → unmanaged.

▲ Exam trap

Only managed packages support publisher upgrades and IP protection. Once an unmanaged package is installed, the publisher can't push updates — the subscriber owns and edits the components. Commercial AppExchange apps are almost always managed.

Modern tooling adds unlocked packages and second-generation (2GP) packaging for source-driven development, but the managed/unmanaged distinction is the exam core.

Worked scenario

An ISV wants to sell an app on the AppExchange, push version upgrades to customers, and keep its source code hidden. Package type?

Reasoning → Answer

A managed package — it's the only type that supports versioned publisher upgrades and IP protection via a namespace. An unmanaged package would expose the source and couldn't be upgraded after install.

◆ Chapter recap
  • Managed = upgradeable, namespaced, IP-protected — for ISV/AppExchange products.
  • Unmanaged = editable template, no upgrades, no IP protection — for open source / metadata moves.
  • Only managed packages support publisher upgrades.
  • Unlocked/2GP packaging exists for source-driven dev, but managed vs unmanaged is the core.
Self-check.
Q1Which package type can the publisher upgrade after install?
Managed. — Unmanaged packages become the subscriber's own and can't be upgraded by the publisher.
Q2You want to share an editable open-source template. Package type?
Unmanaged. — It's fully editable and not IP-protected.
04 · Objective

Determining the deployment plan

Given a scenario, determine the appropriate deployment plan.

Match the mechanism to the org relationship, the change, and whether deletions or Apex are involved. This chapter ties the domain together.

Mechanisms matched to scenarios

SituationBest mechanism
Admin moving config between a sandbox and its production orgChange set
Deploying to an unrelated org / distributing to customersPackage (managed / unmanaged / unlocked)
Source-controlled, developer-driven, CI/CD deploymentSFDX / Metadata API (VS Code, ANT)
Need to delete components in the targetMetadata API (destructiveChanges) — not change sets
Small, one-off declarative changeRebuild by hand or a small change set
◆ Considerations for any plan
  • Test in a sandbox and validate before deploying to production.
  • Apex code coverage — deploying to production needs ≥75% overall coverage, with every trigger having some coverage.
  • Watch dependencies and order (objects before the fields/rules that reference them).
  • Plan data separately — deployment tools move metadata, not records.
▲ Exam trap — the 75% rule

Deploying anything containing Apex to production triggers the 75% code coverage gate — a favourite deployment question even on this declarative-focused exam. And remember change sets can't delete or reach unrelated orgs.

Worked scenario

A consultancy must deploy a completed solution — including some Apex — from their build org into a brand-new client production org they have no sandbox relationship with. Plan?

Reasoning → Answer

The orgs are unrelated, so a change set is out. Distribute via a package (or deploy with the Metadata API / SFDX). Because it contains Apex, the deployment to production must meet the ≥75% code coverage requirement.

◆ Chapter recap
  • Related orgs → change set; unrelated/customers → package; CI/CD → SFDX/Metadata API.
  • Deletions → Metadata API (destructiveChanges), not change sets.
  • Apex to production requires ≥75% code coverage.
  • Always validate first, mind dependencies and order, and plan data separately.
Self-check.
Q1Deploying to an unrelated production org. Change set or package?
A package (or Metadata API / SFDX). — Change sets only work between related orgs.
Q2What coverage is required to deploy Apex to production?
At least 75% code coverage. — Plus every trigger must have some coverage.
Q3How do you delete components in a target org during deployment?
The Metadata API (destructiveChanges). — Change sets can't delete.
