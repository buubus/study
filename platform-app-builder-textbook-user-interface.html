04 · Platform App Builder — Deep Dive 17% of exam

User Interface.

Where the app meets the user: Lightning App Builder assembles the page, page layouts and compact layouts shape the record, and actions replace the old JavaScript buttons. Know which tool owns which surface — and where declarative stops and components begin.

01 · Objective

User interface customisation options

Describe the user interface customisation options — Lightning App Builder, page layouts, dynamic forms, apps, tabs, and list views.

Lightning UI is layered. The App Builder arranges the page; the page layout still governs fields, related lists, and actions; and newer tools (Dynamic Forms) let you place individual fields with visibility rules. The exam tests who owns what.

Lightning App Builder

The declarative canvas for three kinds of page:

  • Record pages — the Lightning page for a record (tabs, related lists, components).
  • App pages — a custom landing page inside a Lightning app.
  • Home pages — the app/home landing experience.

You drag standard, custom, and AppExchange components onto regions, assign pages by app, record type, and profile, and set component visibility with filter rules.

Page layout vs Lightning page — the split

◆ Who controls what
  • Page layout still governs: which fields and related lists appear, layout-level required/read-only, and the mobile & Lightning actions in the action bar.
  • Lightning record page (App Builder) governs the overall component arrangement, tabs, and dynamic behaviour surrounding the record detail.

Dynamic Forms & Dynamic Actions

  • Dynamic Forms — place individual fields and field sections directly on the Lightning page (rather than one monolithic layout block), each with its own visibility rules. Migrates fields off the page layout.
  • Dynamic Actions — show/hide actions based on criteria (show "Escalate" only when Status = Open).
▲ Exam trap

"Show a single field only when another field has a certain value, without code" → Dynamic Forms field visibility (or component visibility). Record types alone only swap whole layouts and picklists — they can't conditionally reveal one field.

Other surfaces

  • Compact layouts — highlights panel / mobile header key fields.
  • List views — filtered, shareable record lists with inline edit and list-view charts.
  • Apps — Lightning apps bundle tabs, a utility bar, and branding; assigned by profile.
  • Tabs — object, web, Visualforce, or Lightning component tabs.
  • Utility bar — persistent footer tools (notes, history, phone).
Worked scenario

On the Opportunity page, the "Discount Reason" field should appear only when Discount > 0, with no code, and reps should also see the account phone in a persistent footer while working. Configure?

Reasoning → Answer

Use Dynamic Forms to place "Discount Reason" with a visibility rule (Discount > 0). Add a utility bar item to the Lightning app for the persistent phone/notes. Neither needs code, and record types can't do the conditional single-field reveal.

◆ Chapter recap
  • App Builder builds record/app/home pages; assign by app, record type, profile; component visibility rules.
  • Page layout owns fields, related lists, and the action bar; the Lightning page owns component arrangement.
  • Dynamic Forms place individual fields with visibility rules — the no-code conditional field tool.
  • Compact layouts, list views, apps, tabs, and the utility bar round out the UI toolkit.
Self-check.
Q1Show one field conditionally on a Lightning page without code. Tool?
Dynamic Forms field visibility (or component visibility). — Record types only swap whole layouts.
Q2Which still controls the actions in the mobile action bar — page layout or Lightning page?
The page layout. — It owns fields, related lists, and actions; the Lightning page owns arrangement.
02 · Objective

Custom buttons, links & actions

Demonstrate the capabilities and use cases for custom buttons, links, and actions.

Three overlapping ways to add interactivity — but in Lightning the modern, mobile-friendly answer is almost always an action, and a key trap is that old JavaScript buttons no longer work.

The three mechanisms

MechanismWhat it isNotes
Custom linkA hyperlink to a URL, record, or external systemRead-only navigation; can pass field values via merge fields
Custom buttonA button on a layout/list view: URL, Visualforce, or (legacy) JavaScriptJavaScript buttons don't work in Lightning — migrate
Action (quick action)Create/update record, log a call, send email, or custom (Flow/LWC/VF)Modern; works in Lightning + mobile; can pre-fill fields
◆ Global vs object-specific actions
  • Object-specific — on one object's layout; can auto-relate and pre-populate from the parent record.
  • Global — available anywhere (publisher, mobile nav); no parent context.
▲ Exam trap — the JavaScript button migration

Migrating from Classic and a JavaScript custom button must keep working in Lightning → it won't. Replace it with a quick action, a Flow launched from an action, or an LWC. This is one of the most common UI scenarios on the exam.

Worked scenario

A Classic org has a JavaScript button that sets several fields and creates a follow-up Task in one click. The org is moving to Lightning. What replaces it?

Reasoning → Answer

JavaScript buttons are unsupported in Lightning. Rebuild the behaviour as a quick action that launches a screen Flow (or an autolaunched Flow from an action) to set the fields and create the Task — declarative and Lightning-native. An LWC is the option if the UI must be bespoke.

◆ Chapter recap
  • Links navigate; buttons can be URL/Visualforce/JS; actions are the modern, mobile-ready choice.
  • JavaScript buttons do NOT work in Lightning — replace with actions, Flows, or LWC.
  • Object-specific actions pre-fill from the parent; global actions have no context.
  • Actions can launch screen flows for guided, no-code interactivity.
Self-check.
Q1A JavaScript custom button must work after moving to Lightning. Options?
Replace it with a quick action, a Flow launched from an action, or an LWC. — JavaScript buttons are unsupported in Lightning.
Q2Which action type pre-populates fields from the parent record?
Object-specific. — Global actions have no parent context.
03 · Objective

Declarative Lightning components

Given a scenario, determine the declarative options available for incorporating Lightning Components in an application.

You can place and configure Lightning components entirely through the App Builder — no code — and you should know which come standard and how to add guided UI without a developer.

Placing components declaratively

In the App Builder you drag components onto a page and configure their properties in the panel. Component sources:

  • Standard components — Highlights Panel, Record Detail, Related Lists, Related Record, Report Chart, Tabs, Accordion, Rich Text, Flow, and more.
  • Custom components — LWC/Aura your developers build and expose to App Builder.
  • AppExchange components — from installed managed packages.
◆ Component visibility

Every component can carry visibility filter rules — show/hide by field values, device, or user — set declaratively. This is how you build responsive, conditional pages without code.

Embedding a Flow with no code

Drop the standard Flow component onto a record or app page and pick a screen flow — a fully declarative way to add guided UI.

▲ Exam trap

"Add interactive, guided functionality to a record page without a developer" → embed a screen Flow via the standard Flow component, or use existing standard/AppExchange components. Custom LWC is only needed when no available component fits.

Worked scenario

Support wants a guided troubleshooting wizard embedded on the Case page, configurable by admins, no developers available. How?

Reasoning → Answer

Build the wizard as a screen Flow and embed it on the Case Lightning page using the standard Flow component in App Builder — optionally with visibility rules so it only appears for certain record types. Entirely declarative.

◆ Chapter recap
  • App Builder places standard, custom, and AppExchange components with no code.
  • Component visibility rules make pages responsive and conditional declaratively.
  • Embed a screen Flow via the standard Flow component for guided UI.
  • Reach for custom LWC only when nothing standard fits.
Self-check.
Q1Add a guided, multi-step wizard to a record page with no developer. Approach?
Build a screen Flow and embed it via the standard Flow component. — Fully declarative.
Q2How do you make a component appear only for certain records?
Component visibility filter rules in App Builder. — Based on field values, device, or user.
04 · Objective

Programmatic (custom) Lightning components

Describe the programmatic customisations available for incorporating custom Lightning Components in an application.

You won't write components for this exam, but you must know what custom LWC/Aura are for and how they surface to admins — because the tested idea is the boundary between declarative and code.

LWC and Aura

  • Lightning Web Components (LWC) — the modern, standards-based framework (HTML + modern JavaScript). Preferred for new work.
  • Aura components — the older framework; still supported and can host LWC.
◆ When custom components are justified

Reach for custom Lightning components only when standard and AppExchange components, plus declarative tools, cannot deliver the required interface or behaviour — bespoke UI, complex client-side interactivity, custom visualisations.

How developers expose them to admins

  • A component's metadata configuration (targets) determines where it can be placed — record page, app page, home page, utility bar, Flow screen, etc.
  • Design attributes / properties let admins configure it in App Builder without touching code.
  • Once exposed, placing it feels identical to placing a standard component.
▲ Exam trap

The exam tests the boundary: prefer declarative; use custom LWC only when nothing standard fits. And a custom component must be configured by the developer with the correct targets before an admin can drag it onto a page.

Worked scenario

A team asks for an interactive map that plots nearby accounts and lets users drag pins — nothing standard does this. What's the right recommendation, and what must the developer do?

Reasoning → Answer

This bespoke interactivity is beyond declarative and standard components, so a custom LWC is justified. The developer must expose it with the correct targets (e.g. record/app page) and design properties so admins can add and configure it in App Builder.

◆ Chapter recap
  • LWC is the modern framework; Aura is legacy and can host LWC.
  • Use custom components only when declarative and standard/AppExchange components can't deliver.
  • Developers expose components via targets and design attributes so admins configure them in App Builder.
  • The tested idea is the declarative-vs-code boundary.
Self-check.
Q1When is a custom LWC the right answer on this exam?
Only when no standard/AppExchange component or declarative tool can meet the requirement. — Prefer declarative first.
Q2What must a developer set so an admin can place a custom component in App Builder?
The component's targets (and design attributes). — Targets define where it can be dropped.
